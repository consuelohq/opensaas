# Get Started

Consuelo has two primary product paths today: **OS** and **Dialer**.

Use OS when you want agents to work through a safe portal into your company. Use Dialer when you want to set up calls, queues, caller ID, coaching, and revenue workflows inside the Consuelo app.

This page helps you choose the right path before you read deeper docs.

## Choose your path

If you want to... | Start with | Why
--- | --- | ---
Connect an agent to company context | [Install Consuelo OS locally](/get-started/install-local) | OS gives agents a safe portal into the workspace.
Let an agent run approved workflows | [Run your first skill](/get-started/first-skill) | Skills are the repeatable workflows agents can run.
Create tracked reports or generated files | [Create your first artifact](/get-started/first-artifact) | Artifacts keep outputs visible, auditable, and tied to executions.
Decide whether to run locally or in the cloud | [Local OS and Cloud OS](/get-started/local-vs-cloud) | Local and cloud use the same model with different storage and sharing choices.
Set up calling and queue workflows | Dialer docs | Dialer is the calling product surface.

## OS path

Consuelo OS turns your company into an agent-ready workspace.

The first OS setup path is local: install OS on your machine, connect your agents, run a skill, and create a tracked artifact. Local OS is useful for founders, internal builders, and solo users. Cloud OS is for team workspaces that need shared access, hosted storage, and hosted observability.

```text
Install locally
  -> connect agents
  -> run first skill
  -> create first artifact
  -> choose local or cloud storage
```

## Dialer path

Dialer is for calls, queues, transfers, caller ID, coaching, and sales workflows inside the Consuelo app.

If you are setting up a calling team, start with the Dialer tab. The OS docs may still be useful later when agents need to summarize calls, generate follow-ups, or create artifacts from sales activity.

## What is shared between OS and Dialer

OS and Dialer are different product surfaces, but they should eventually share a workspace model:

- people and companies
- calls and call outcomes
- notes and tasks
- artifacts and reports
- approvals and audit trails
- integrations and capability health

That shared workspace model is why OS docs use terms like structured data, artifacts, and capabilities instead of talking only about local files or raw scripts.
