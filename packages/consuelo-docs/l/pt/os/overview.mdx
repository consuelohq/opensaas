# Consuelo OS

Consuelo OS turns your company into an agent-ready workspace.

Use this page to choose the right OS surface before exploring deeper docs. The portal is the agent entry point, skills are approved workflows, scripts are the implementation behind skills, artifacts are tracked outputs, approvals are human decision points, and capabilities are the connected systems OS can use.

This is an overview and routing page. For deeper behavior, use [How Consuelo OS works](/os/how-it-works) and the concept pages linked below.

## Start here

For most teams, start with the OS portal, then move into skills and artifacts. Add capabilities only when a skill needs them.

## Choose the right OS surface

### 1. Use the portal when the agent needs to enter the workspace

The OS portal is the boundary between an agent and the company. It tells the agent what workspace it is in, what skills are available, which capabilities are connected, what rules apply, and how to run approved work.

Agents should enter through the portal instead of improvising direct access to files, app internals, databases, provider settings, or integration credentials.

Read next: [OS portal](/os/concepts/portal)

### 2. Use a skill when the agent needs an approved workflow

A skill is the product unit of work. It defines what the agent can do, what input it needs, which capability must be connected, what output it returns, and when a human approval is required.

Use skills for repeatable work such as daily revenue briefs, call coaching notes, follow-up drafts, campaign review, weekly reports, or landing page drafts.

Read next: [Skills](/os/concepts/skills) and [Run your first skill](/get-started/first-skill)

### 3. Use a script when the workflow needs implementation

Scripts are executable implementation behind skills. Builders maintain scripts. Agents call skills. OS connects the two.

Use scripts for focused execution: fetch a workspace snapshot, transform records, generate a report, write an artifact, or prepare an approval request. Scripts should return structured results that OS can record.

Read next: [Scripts](/os/concepts/scripts)

### 4. Use an artifact when output must be durable

A file is bytes. An artifact is the product record around those bytes.

Use artifacts for generated outputs a person may need later: reports, briefs, follow-up drafts, campaign drafts, landing page drafts, exports, documents, or app-native generated records. Artifacts connect the output to the skill execution, source records, storage location, version, status, and trace ID.

Read next: [Files and artifacts](/os/concepts/files-and-artifacts) and [Create your first artifact](/get-started/first-artifact)

### 5. Use approvals when the agent prepares high-impact work

Approvals let OS pause before a skill publishes, sends, updates, spends, or changes shared state.

The skill prepares the proposed action. OS records the approval request. A human approves or rejects. The execution continues or stops with the decision recorded.

Read next: [Approvals](/os/concepts/approvals)

### 6. Use capabilities when a skill needs a connected system

A capability is something OS can use: local storage, workspace data, cloud artifact storage, Twilio, Google Ads, Meta Ads, GHL, Stripe, Slack, Discord, Sentry, PostHog, or another connected surface.

Capabilities should be added when skills need them. A missing capability should disable the specific skill that depends on it and return a clear status, rather than making all of OS feel broken.

Read next: [Integrations and capabilities](/os/concepts/integrations-and-capabilities)

## Core architecture

The high-level OS loop is intentionally small:

```text
agent
  -> OS portal
    -> workspace context
    -> approved skill
      -> script implementation
        -> structured data and capabilities
        -> artifacts
        -> approvals when needed
        -> execution logs and trace IDs
```

The agent brings reasoning. OS brings the workspace boundary, workflow contract, storage model, and audit trail.

## Local OS and Cloud OS

Consuelo OS can run locally or in the cloud.

Local OS is for fast setup, local agents, founder workflows, internal builders, and solo use. It should install on a machine, create an OS home, connect agents, store metadata in SQLite, and keep artifact bytes locally.

Cloud OS is for team workspaces and customer deployments. It should use the app for identity, shared artifacts, managed capabilities, hosted observability, and team approval surfaces.

Both modes should use the same product model. A skill should create an artifact through the artifact layer. The artifact layer decides whether bytes are stored locally or in cloud storage.

Read next: [Local OS and Cloud OS](/get-started/local-vs-cloud)

## Configure access and approvals

OS access should be governed before a skill runs. The agent should only see the skills and capabilities that survive workspace settings, runtime mode, capability health, user permissions, and approval rules.

Use this order when diagnosing access:

1. Confirm the agent is connected to the OS portal.
2. Confirm the skill is registered and visible.
3. Confirm required capabilities are connected and healthy.
4. Confirm the user and workspace can run the skill.
5. Confirm the skill input matches its contract.
6. Confirm the action has the required approval state.
7. Confirm the execution produced a trace ID.

Read next: [OS portal](/os/concepts/portal), [Integrations and capabilities](/os/concepts/integrations-and-capabilities), and [Approvals](/os/concepts/approvals)

## Extend OS

Choose the extension path by the job you need OS to do:

Need | Extension path
--- | ---
A new repeatable workflow | Add a skill.
A new executable implementation | Add or update the script behind a skill.
A new generated output type | Add an artifact type or artifact metadata behavior.
A new external system | Add a capability and wire skills to it.
A new safety checkpoint | Add an approval policy or approval delivery surface.
A new data shape | Add an app/API facade or artifact-backed concept.
A new team runtime | Move the workflow from local OS to Cloud OS storage and observability.

Start with the product contract first: what the agent sees, what input the workflow accepts, what output OS records, and what must be approved.

## Troubleshoot missing skills or capabilities

If an agent cannot see or run a skill, start with the effective OS surface for the current run:

Check | What it tells you
--- | ---
Agent connection | Whether the agent can reach the portal.
Portal context | Whether OS is returning workspace identity and visible skills.
Skill registration | Whether the workflow exists in the current runtime.
Capability health | Whether required systems are connected and healthy.
Runtime mode | Whether the skill is local-only, cloud-only, or available in both.
User/workspace permission | Whether this user can run the skill in this workspace.
Approval state | Whether the skill paused for a human decision.
Execution trace | Whether OS recorded what happened.
Artifact record | Whether the expected output was stored durably.

Most failures should return product statuses such as `missing_capability`, `not_configured`, `validation_failed`, `approval_required`, `permission_denied`, or `execution_failed`.
