## Overview

Computer respects your existing permission structure. It can only access and modify data that your role allows — no exceptions.

## How Permissions Work

### Role-Based Access

Computer operates under the same role system as users:

| Role Level | What Computer Can Do                       |
| ---------- | ------------------------------------------ |
| Admin      | Access all data, modify all records        |
| Manager    | Access team data, modify within team       |
| Member     | Access own data, limited modifications     |
| Custom     | Depends on specific permissions configured |

### Assigning a Role to Computer

To configure Computer's access:

1. Go to **Settings → Roles**
2. Select the role you want Computer to use
3. Click **Assignment** tab
4. Under **AI Agents**, click **+ Assign AI agent**
5. Select which Computer instance to assign
6. Save changes

<Note>
  Each workspace can have multiple Computer instances with different roles. For
  example, a "Sales Computer" with access to deals, and a "Support Computer"
  with access to tickets.
</Note>

## Permission Scope

### Object-Level Permissions

Control which objects Computer can access:

| Permission   | Can Access                       | Cannot Access        |
| ------------ | -------------------------------- | -------------------- |
| Read         | Search, view, analyze            | Edit, create, delete |
| Read & Write | All operations                   | None for that object |
| No Access    | Cannot see, search, or reference | Completely hidden    |

**Example:**

If Computer's role has "No Access" to Companies:

- Cannot search for companies
- Cannot view company details
- Cannot see company name in contact records
- Coaching won't include company context

### Field-Level Permissions

Even within an accessible object, some fields may be hidden:

| Field Permission | Computer Behavior               |
| ---------------- | ------------------------------- |
| Visible          | Can read and use in suggestions |
| Hidden           | Cannot see or reference         |
| Read-Only        | Can see but cannot modify       |
| Editable         | Full read/write access          |

**Example:**

If "Salary" field is hidden:

- Computer won't mention salary in coaching
- Cannot search by salary
- Won't appear in analysis

## What Each Tool Requires

| Tool            | Minimum Permissions      |
| --------------- | ------------------------ |
| Search Contacts | Read on People           |
| Get Contact     | Read on People           |
| List Deals      | Read on Opportunities    |
| Log Call        | Write on People (notes)  |
| Update Deal     | Write on Opportunities   |
| Create Note     | Write on People          |
| Create Task     | Write on Tasks           |
| Add to Queue    | Read on People           |
| Get Analytics   | Read on relevant objects |
| Search KB       | Read on Knowledge Base   |

## Why This Matters

### Security

Limit Computer's access to only what's needed for its role. A sales Computer doesn't need access to HR records.

### Compliance

Ensure Computer only processes data it needs — important for GDPR, SOC 2, and other compliance frameworks.

### Control

Prevent unintended modifications. You can give Computer read-only access to sensitive data.

### Auditability

Every action Computer takes is logged with which agent performed it. You can trace exactly what was modified.

<Note>
  For Computer running in workflows, role assignment ensures it cannot access or
  modify data outside its intended scope — even if the workflow has broader
  permissions.
</Note>

## Common Configurations

### Sales Team Computer

```
Role: Sales Agent
├─ People: Read & Write
├─ Companies: Read & Write
├─ Opportunities: Read & Write
├─ Tasks: Read & Write
├─ Notes: Read & Write
└─ Other objects: No Access
```

**What it can do:**

- Search and manage contacts
- Update deal stages
- Log calls and notes
- Create follow-up tasks

**What it can't do:**

- Access billing information
- Modify workspace settings
- View other teams' data

### Read-Only Computer

```
Role: Analytics Viewer
├─ People: Read
├─ Companies: Read
├─ Opportunities: Read
└─ Other objects: No Access
```

**What it can do:**

- Answer questions about pipeline
- Generate reports and analytics
- Provide coaching insights

**What it can't do:**

- Create or modify any records
- Log calls or notes
- Create tasks

### Full Admin Computer

```
Role: Admin
├─ All objects: Read & Write
└─ Settings: Access
```

**What it can do:**

- Everything
- Modify configuration
- Manage other users

**When to use:**

- Trusted automation
- Internal tooling
- Development/testing

## Auditing Computer Actions

Every action Computer takes creates an audit trail:

1. Go to the record (contact, deal, etc.)
2. Scroll to **Activity** or **History**
3. Actions are marked with "Computer" or agent name
4. Click to see full details of what changed

## Best Practices

### Principle of Least Privilege

Start with minimal access and add permissions as needed. Don't give write access unless Computer actively modifies records.

### Separate Roles for Different Uses

Create specific roles for specific purposes:

- `computer-sales`: For sales coaching and deal management
- `computer-analytics`: For reporting and insights
- `computer-admin`: For automation and workflows

### Regular Audits

Review Computer's actions periodically:

- Are the modifications appropriate?
- Any unexpected access patterns?
- Should permissions be tightened?

### Test Before Deploying

When configuring a new Computer role:

1. Test with sample data
2. Verify the suggestions are appropriate
3. Confirm it can't access restricted data
4. Deploy to production
