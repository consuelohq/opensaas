> **Provider-error classification is both too broad and silently drops observability.**
> 
> Two concerns with `isProviderCustomerPhoneFailure` and how it's used in the outer `catch`:
> 
> 1. `message.includes('invalid phone number')` is very generic. Twilio (and other providers) use similar wording for caller-id / from-number validation failures too, so a misconfigured `TWILIO_DEFAULT_NUMBER` or account-level from-number issue will be mis-classified as "Invalid customer phone number" and returned as a 400 to the end user, masking a real operator problem. Consider anchoring the match to Twilio's actual error surfaces (e.g., error codes like `21215` geo-permissions, `21211` invalid 'To' number, `13227` dial geographic permission) when the underlying error object exposes them, and keep the substring check as a last-resort fallback.
> 
> 2. Once the error is converted to `BadRequestException` at line 206, it short-circuits the outer error path and therefore skips both `this.logger.error(...)` and `Sentry.captureException(...)`. A genuine provider outage that happens to match one of these substrings will now produce 400s with zero telemetry. Recommend emitting at least a `logger.warn` (and optionally a Sentry breadcrumb) before throwing, so on-call can still correlate spikes.
